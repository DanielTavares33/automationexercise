# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies
npm ci

# Install Playwright browsers (required before first run)
npx playwright install chromium

# Run all tests (4 workers by default)
npm test

# Run by tag group
npm run test:smoke       # @smoke scenarios — fast CI gate
npm run test:regression  # full suite
npm run test:login       # login feature only
npm run test:register    # register feature only

# Run with custom workers or headless
WORKERS=8 npm test
HEADLESS=true npm test
WORKERS=4 HEADLESS=true npm run test:smoke
```

`npm test` runs `bddgen` first (auto-generates `.features-gen/` from feature files), then `playwright test`. Never run `playwright test` alone without `bddgen`.

## Architecture

`playwright-bdd` + `@playwright/test` in TypeScript, targeting [automationexercise.com](https://automationexercise.com). Feature files are Gherkin; the test runner is Playwright (not Cucumber). `playwright.config.ts` is the single config file.

### Layer responsibilities

- **`features/`** — Gherkin `.feature` files organized by domain. Each scenario has two tag dimensions: execution tier (`@smoke`/`@regression`) and outcome (`@positive`/`@negative`), plus domain tag (`@login`, `@register`).
- **`locators/<page>.locators.ts`** — One class per page with `readonly Locator` fields initialized in the constructor. CSS selectors only (e.g. `[data-qa="..."]`, `p[style="color: red;"]`). Never use `{ hasText }` options or role-based selectors.
- **`pages/<page>.page.ts`** — Page Object Models extending `BasePage`. Constructor instantiates the locator class and receives `Page`. Method order: constructor → public → private.
- **`support/fixtures.ts`** — Extends `playwright-bdd`'s `test` with custom fixtures (`homePage`, `loginPage`, `registerPage`). Each fixture creates a new page object instance per test. This is the integration point between Playwright's fixture system and the page objects.
- **`step_definitions/<domain>.steps.ts`** — Use `createBdd(test)` from `playwright-bdd` (importing `test` from `support/fixtures.ts`). Steps receive destructured fixtures: `async ({ homePage, loginPage, ... }) => { ... }`. No `this` context, no World class.
- **`.features-gen/`** — Auto-generated by `bddgen`, gitignored. Do not edit.

### Key patterns

**Adding a new page:** create `locators/<page>.locators.ts` → `pages/<page>.page.ts` → add fixture to `support/fixtures.ts` → create `step_definitions/<page>.steps.ts`.

**GDPR consent popup:** `HomePage.visitHomePage()` calls `googleConsentPopup()` which clicks the consent button if visible. Any step that navigates to the site fresh should call `visitHomePage()` (via the `Given I visit the home page` step, defined in `home.steps.ts`).

**Shared step dispatch:** Generic steps like `When I click on the {string} button` live in `home.steps.ts` and switch on the button text, delegating to the appropriate page object. Add new button cases to the switch rather than creating duplicate step definitions.

**faker:** Used in `RegisterPage` and `LoginPage` for generating random test data (`@faker-js/faker`).

## Workflow Rules

- **Do not run tests after edits.** Never run `npm test` or any test command automatically after making code changes. Only run tests when explicitly asked.
